## Project Overview

Toterninger is an interactive two-dice roller web app, primarily designed for Settlers of Catan. It supports solo play with persistent stats and a Firestore-backed multiplayer mode (host + joiners over a shared session code). Features include animated CSS dice, shake-to-roll on mobile, procedural Web Audio sound, contextual roll commentary, and a Catan mode that highlights when 7 (the robber) is rolled.

## Commands

- `npm run dev` — Start dev server (Vite)
- `npm run build` — Type-check with `vue-tsc -b` then build for production
- `npm run preview` — Preview the production build locally

There are no tests or linters configured.

## Architecture

- **Framework:** Vue 3 with Composition API and `<script setup>` SFCs
- **Language:** TypeScript (strict mode via `tsconfig.app.json`)
- **Bundler:** Vite 8 with `@vitejs/plugin-vue`
- **Backend:** Firebase Firestore (config inlined in `src/firebase.ts`) for multiplayer sessions only
- **Entry point:** `index.html` → `src/main.ts` → `src/App.vue`
- **UI language:** Norwegian (bokmål)

### Routing

Hash-based mini-router in `src/router.ts` — no vue-router. Routes:
- `''` / unmatched → `{ name: 'home' }` (solo mode + 4-tab nav)
- `#/session/XXX-XXX-XXX` → `{ name: 'session', sessionId }` (multiplayer)
- `#/session/new` → handled in `App.vue` as a special `showCreateFlow` flag (mounts `MultiplayerGame` with `sessionId="__new__"` so it shows the name-input/create flow before a session exists)

Session IDs are 9-digit `NNN-NNN-NNN` strings generated by `generateSessionId()`.

### Solo vs. multiplayer: shared components, swapped backend

`DiceRoller.vue` and `Statistics.vue` are reused in both solo and multiplayer modes. They consume a `StatisticsAPI` (defined in `useStatistics.ts`) via `provide`/`inject` using `statisticsKey`:

- **Solo** (`App.vue` → `DiceRoller`/`Statistics`): backed by `useStatistics()` — localStorage-persisted reactive singleton.
- **Multiplayer** (`MultiplayerGame.vue`): backed by `useMultiplayerStatistics(session, submitRoll, resetStats)` which adapts the live Firestore session document to the same `StatisticsAPI` shape, then `provide`s it.

When changing the stats interface, update both implementations and the `StatisticsAPI` type in `useStatistics.ts`.

`DiceRoller` distinguishes solo vs multiplayer at runtime by whether `inject(statisticsKey)` returned anything. In multiplayer it hydrates from the latest roll on mount and watches `stats.stats.rolls.length` to mirror other players' rolls (animated for a single new roll, silent for initial load / bulk catch-up). It skips its own rolls (matched by `playerId`) since `roll()` already animates them locally.

### Multiplayer (Firestore)

`useSession.ts` is the single source of multiplayer logic. One Firestore document per session at `sessions/{sessionId}` matching `SessionDocument`:
- Host (first player to create) is identified by `hostId` and is the only one allowed to start/restart/reset/remove players.
- `players[]` is an ordered list; turns advance round-robin via `currentPlayerIndex`.
- `heartbeats: Record<playerId, timestamp>` — each client writes its own heartbeat every 5s; players with no heartbeat in 15s are considered "away".
- `turnTimeout: number | 'manual' | 'admin'` is **snapshotted onto the session document from the host's settings at create time** (not read from each client's local settings). All clients agree on the mode.
  - numeric → `turnStartedAt` + the value drive the countdown; **the host's client auto-advances** when it hits 0 (`advanceTurn`)
  - `'manual'` → no timer; the active player presses "Gi turen videre"
  - `'admin'` → no timer; only the host presses "Neste spiller"
  - The `canPassTurn` computed encapsulates who's allowed to press the button; `passTurn()` re-checks it before writing
- Auto-advance on roll **only applies in timer mode**. In `'manual'` / `'admin'` the turn is held after rolling — `submitRoll` writes the roll but not `currentPlayerIndex`, and the turn advances only when someone presses the "Gi turen videre" / "Neste spiller" button.
- Player identity is a UUID stored in localStorage by `usePlayerId.ts` — same browser = same player across sessions.
- **Reconnect**: `src/lastSession.ts` persists `{ sessionId, name }` to localStorage on `createSession`/`joinSession`. `App.vue` checks it on mount of the home view: one-shot `getDoc` against the saved session — if it exists, status is `lobby`/`playing`, AND the local `playerId` is still in `players[]`, a banner offers to rejoin. Otherwise the saved session is silently cleared. The "Tilbake" button in `MultiplayerGame` does NOT clear the saved session — by design it's the user's path back to the same game via the reconnect banner.

There are currently no Firestore security rules in this repo; the API key is committed (public web-app key, gated by Firestore rules in the Firebase console).

### Composables

- `useSettings` — reactive singleton, localStorage-persisted, applies `<html data-theme>` for theming. Includes a migration shim from the old `catanMode: boolean` to `gameMode: 'standard' | 'catan' | 'catan-ck'`.
- `useStatistics` — solo stats singleton + the canonical `StatisticsAPI` shape and `statisticsKey` injection key.
- `useMultiplayerStatistics` — adapts a Firestore session into a `StatisticsAPI`.
- `useShake` — DeviceMotion shake detection with iOS 13+ permission flow; must be cycled on page load (see commit `36624b4`) for browser to register the listener reliably.
- `useSound` — Web Audio API procedural dice sounds; no audio files.
- `useSession` / `usePlayerId` — multiplayer (see above).

### Game modes

`settings.gameMode`:
- `standard` — plain two dice
- `catan` (default) — sum displayed prominently; rolling 7 triggers a pulsing red "Flytt røveren!" alarm; 7 is highlighted red in stats
- `catan-ck` — Cities & Knights variant; renders an additional `EventDie.vue` (ship/barbar + colored castle for science/politics/trade)

### Other UI notes

- Single-file CSS dice via `DiceFace.vue` using CSS Grid (no images/canvas)
- Themes (`light` / `dark` / `neon`) are applied to `document.documentElement.dataset.theme` by `useSettings`; component CSS uses `var(--*)` tokens
- Tabs (solo): Kast / Statistikk / Innstillinger / Om — switched via `v-if`, no router
